# Background Jobs — Фоновые процессы и планировщик

## 1. Назначение

Фоновые задачи используются для выполнения процессов, которые:

- не должны блокировать HTTP-запрос
- выполняются по расписанию
- требуют повторной проверки состояния
- связаны с уведомлениями

Бизнес-логика фоновых задач находится только в backend.

—

## 2. Архитектурный принцип

Фоновые задачи:

- не содержат бизнес-логики
- вызывают сервисный слой
- работают через dependency injection
- используют async

—

## 3. Текущая реализация (MVP)

На этапе MVP используется APScheduler внутри backend.

Причины:
- простота
- нет отдельного брокера
- минимальная сложность

Запуск scheduler происходит при старте приложения.

—

## 4. Будущая реализация (Production)

При росте нагрузки:
- Celery + Redis
- Celery Beat для расписания
- отдельный worker-контейнер

Backend не должен зависеть от конкретной реализации.
Сервисный слой должен быть абстрагирован.

—

## 5. Основные фоновые процессы

### 5.1 Проверка незакрытых смен
Запуск: каждые 10 минут.

Алгоритм:
1. Найти смены:
   - end_time IS NULL
   - start_time < now - 12.5 часов
2. Если уведомление не отправлено:
   - отправить Telegram уведомление
   - записать в notification_logs
3. Если прошло 14 часов:
   - закрыть смену
   - выставить 12.5 часов
   - отметить is_auto_closed = true
   - создать audit запись

### 5.2 Очистка логов
Запуск: 1 раз в сутки.

Удаляются:
- некритические логи старше 30 дней

Не удаляются:
- audit_logs
- критические ошибки

### 5.3 Перерасчет зарплаты (опционально)
Запуск:
- по требованию
- после изменения ставок
- после корректировки смен

Перерасчет выполняется через PayrollService.

### 5.4 Проверка доставки уведомлений
Если Telegram API вернул ошибку:
- статус сохраняется
- owner уведомляется при критической ошибке

Повторные уведомления не отправляются.

—

## 6. Идемпотентность

Каждая фоновая задача должна быть идемпотентной.

Пример:
- повторный запуск автозакрытия не должен закрывать смену дважды.
- повторный запуск очистки не должен удалять активные записи.

—

## 7. Логирование

Каждая фоновая операция обязана логировать:
- organization_id
- тип задачи
- количество обработанных записей
- ошибки

—

## 8. Ограничения

Запрещено:
- выполнять тяжелые расчеты в HTTP endpoint
- выполнять фоновые задачи через “хаки”
- писать бизнес-логику внутри scheduler

—

## 9. Масштабирование

При переходе на Celery:
- каждый тип задачи может стать отдельным worker
- можно масштабировать payroll отдельно от shift-monitoring
- можно запускать workers для отдельных филиалов

—

## 10. Инварианты

- фоновые процессы не нарушают tenant isolation
- все изменения проходят через сервисный слой
- нет прямого доступа к БД вне сервисов
