# Notifications Architecture — Архитектура уведомлений

## 1. Назначение

Система уведомлений предназначена для:
- напоминаний сотрудникам
- информирования о критических событиях
- уведомления о расчетах зарплаты
- информирования владельца о сбоях

Уведомления не являются источником бизнес-логики.
Они только отражают события, произошедшие в системе.

—

## 2. Каналы уведомлений

На текущем этапе используется:
- Telegram Bot (основной канал)

В будущем возможны:
- Email
- Push
- SMS

Архитектура должна позволять расширение каналов.

—

## 3. Архитектурный принцип

Уведомления:
- инициируются сервисным слоем
- отправляются через NotificationService
- логируются в notification_logs
- не содержат бизнес-логики

Backend отвечает за:
- решение, нужно ли отправлять уведомление
- формирование текста
- контроль частоты

Telegram Bot:
- только транспорт
- не принимает решений

—

## 4. Основные типы уведомлений

### 4.1 Напоминание о закрытии смены
Условие:
- прошло 12.5 часов
- смена не закрыта

Получатель:
- сотрудник

Частота:
- 1 раз на смену

### 4.2 Автозакрытие смены
Условие:
- прошло 14 часов
- смена не закрыта

Получатель:
- сотрудник

Частота:
- 1 раз

### 4.3 Превышение процента при расчете зарплаты
Условие:
- суммарный процент превышает допустимый порог

Получатель:
- owner (организационный или глобальный)

Частота:
- 1 раз на расчет

### 4.4 Критическая ошибка системы
Условие:
- сбой автозакрытия
- сбой расчета зарплаты
- сбой отправки уведомления

Получатель:
- owner / superadmin

Частота:
- 1 раз на событие

—

## 5. NotificationService

NotificationService отвечает за:
1. проверку, отправлялось ли уведомление ранее
2. запись в notification_logs
3. отправку сообщения через Telegram Bot API
4. обработку ошибок

—

## 6. Защита от дублей

Для предотвращения спама:
- в notification_logs хранится уникальный ключ события:
  (entity_type + entity_id + notification_type)

Перед отправкой проверяется:
- не существует ли уже запись с таким ключом

Если существует — повтор не отправляется.

—

## 7. Структура notification_logs

Поля:
- organization_id
- user_id
- entity_type
- entity_id
- notification_type
- status (sent / failed)
- message_preview
- created_at

—

## 8. Локализация

Тексты уведомлений формируются на backend.

В будущем возможно:
- хранение шаблонов
- поддержка нескольких языков

—

## 9. Ограничения

Запрещено:
- отправлять уведомления напрямую из scheduler без сервиса
- делать логику проверки внутри Telegram-бота
- отправлять одно и то же уведомление несколько раз
- делать тяжелые вычисления во время отправки

—

## 10. Масштабирование

При переходе на Celery:
- отправка уведомлений может быть вынесена в отдельный worker
- можно разделить типы уведомлений по очередям
- можно реализовать retry-механизм

Архитектура должна позволять:
- смену транспортного канала
- добавление новых типов уведомлений
- централизованное управление частотой
