# Tenant Isolation — Изоляция филиалов и уровни доступа

## 1. Общий принцип

Система Clear Barn поддерживает работу:

- одного пользователя в нескольких филиалах
- разных ролей пользователя в разных филиалах
- глобального Owner и SuperAdmin

При этом сохраняется строгая изоляция данных между филиалами.

Нарушение tenant isolation считается критической архитектурной ошибкой.

—

## 2. Уровни доступа

В системе существуют два уровня доступа:

### 2.1 Глобальный уровень (System Scope)

Определяется полем global_role в таблице users.

Возможные значения:
- superadmin — полный контроль всей системы
- owner — глобальный владелец сети
- none — обычный пользователь

Глобальный уровень не отменяет организационную изоляцию, а расширяет доступ к выбору филиалов.

—

### 2.2 Организационный уровень (Organization Scope)

Определяется таблицей user_organizations:

- user_id
- organization_id
- role (owner | manager | employee)

Роль действует строго в рамках конкретной организации.

—

## 3. Разделение сущностей

В системе разделяются следующие понятия:

### User
Физический человек.

### UserOrganization
Роль пользователя в конкретном филиале.

### Employment
Конкретная работа пользователя в филиале.
Может включать:
- должность
- почасовую ставку
- процент

Один пользователь может иметь несколько employments в разных филиалах.

—

## 4. Основное правило изоляции

Каждая бизнес-сущность обязана содержать поле:

    organization_id

Это касается:
- employments
- shifts
- payroll_records
- bonuses_penalties
- audit_logs
- notification_logs
- configuration

—

## 5. Источник organization context

organization_id НЕ принимается из body запроса.

organization context определяется backend’ом:

1. По выбранному активному филиалу пользователя
2. По связи user_organizations
3. Для Owner — через явный выбор филиала или глобальный режим

—

## 6. Контекст доступа (Access Scope)

При каждом запросе backend определяет:

### Organization Scope
Используется по умолчанию.
Все данные фильтруются по текущему organization_id.

### Global Scope
Доступен только:
- superadmin
- owner

Global scope позволяет:
- агрегировать данные по нескольким филиалам
- формировать глобальные отчёты
- получать суммарную зарплату пользователя по всем филиалам

Но даже в global scope данные остаются привязанными к organization_id.

—

## 7. Обязательная фильтрация

В organization scope:

- Все SELECT обязаны фильтроваться по organization_id
- Проверяется принадлежность сущности организации

В global scope:

- Разрешены агрегирующие запросы
- Запрещено изменение данных без указания organization context

—

## 8. Запрещенные паттерны

❌ Передача organization_id в теле запроса  
❌ Хардкод organization_id  
❌ Глобальные SELECT без проверки scope  
❌ Кросс-организационные JOIN без явной логики  
❌ Использование одного organization_id для всех ролей  

—

## 9. Инварианты безопасности

1. Пользователь никогда не может видеть данные организации, к которой не привязан.
2. Manager видит только свой филиал (в рамках выбранного контекста).
3. Employee видит только свои данные.
4. Owner может агрегировать данные по сети (global scope).
5. Tenant isolation сохраняется даже при global scope.

—

## 10. Проверка нового endpoint

Перед добавлением endpoint необходимо проверить:

- Определяется ли access scope?
- Применяется ли фильтрация по organization_id?
- Проверяется ли принадлежность сущности?
- Не нарушает ли endpoint изоляцию?

Если хотя бы один пункт нарушен — endpoint небезопасен.
