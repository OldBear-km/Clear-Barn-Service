# RBAC — Модель ролей и контроль доступа

## 1. Общий принцип

В системе используется двухуровневая модель ролей:

1. Глобальная роль (system-level)
2. Организационная роль (organization-level)

Все проверки доступа выполняются исключительно на Backend.

Frontend выполняет только визуальные ограничения.

—

## 2. Глобальные роли (global_role)

Хранятся в таблице users.

Возможные значения:

- superadmin
- owner
- none

### superadmin
- Полный доступ ко всем организациям
- Может выполнять любые действия
- Может работать в global scope

### owner
- Глобальный владелец сети
- Может просматривать и агрегировать данные всех филиалов
- Может управлять организациями
- Может работать в global scope

### none
- Обычный пользователь
- Доступ определяется только организационными ролями

—

## 3. Организационные роли (organization role)

Хранятся в таблице user_organizations.

Возможные значения:

- owner
- manager
- employee

Роль действует строго в рамках конкретной организации.

—

## 4. Приоритет ролей

Порядок проверки:

1. Проверяется global_role
2. Если нет достаточного доступа — проверяется organization role
3. Если ни одна роль не даёт доступ — возвращается 403

—

## 5. Scope модели

### Organization Scope (по умолчанию)

- Все действия происходят в рамках выбранного organization_id
- Данные фильтруются строго по organization_id

### Global Scope

Доступен только:

- superadmin
- owner (global_role)

Используется для:

- агрегированных отчетов
- суммарного payroll по сети
- системного управления

Global scope не позволяет изменять данные без указания конкретной организации.

—

## 6. Матрица доступа (упрощённая)

### SuperAdmin
- Чтение: всё
- Изменение: всё
- Удаление: всё
- Global scope: да

### Global Owner
- Чтение: все организации
- Изменение: в рамках organization scope
- Удаление: в рамках organization scope
- Global scope: да

### Organization Owner
- Чтение: свой филиал
- Изменение: свой филиал
- Удаление: свой филиал
- Global scope: нет

### Manager
- Чтение: свой филиал
- Изменение: смены и часы сотрудников
- Удаление: запрещено (если не указано иначе)
- Global scope: нет

### Employee
- Чтение: свои данные
- Изменение: свои смены (до формирования отчета)
- Удаление: запрещено
- Global scope: нет

—

## 7. Алгоритм проверки доступа (логика backend)

1. Определить access scope
2. Проверить global_role
3. Проверить наличие user_organization для текущего organization_id
4. Проверить разрешение на действие
5. Проверить принадлежность сущности пользователю (если требуется)

Если любой шаг не проходит — вернуть 403 Forbidden.

—

## 8. Инварианты безопасности

- Нельзя доверять роли, переданной клиентом.
- Нельзя доверять organization_id из body запроса.
- Нельзя пропускать проверку принадлежности сущности.
- Все проверки выполняются в сервисном слое.

—

## 9. Специальные правила

- Если Manager редактировал смену сотрудника — сотрудник теряет право её редактировать.
- Автозакрытие смены не зависит от ролей.
- Бонусы и штрафы может изменять только owner (организационный или глобальный).

—

## 10. Расширяемость

Модель должна позволять:

- Добавление новых ролей
- Добавление новых разрешений
- Введение permission-based системы в будущем

Текущая реализация — role-based с возможностью эволюции.
