# Security Architecture — Архитектура безопасности

## 1. Общий принцип

Безопасность реализуется исключительно на Backend.

Frontend (Mini App, Web Admin) выполняет только визуальные ограничения.

Никакая логика доступа не может быть реализована только на клиенте.

—

## 2. Аутентификация

### 2.1 Web Admin

- Авторизация по email + пароль
- Пароли хранятся в виде хеша (bcrypt или argon2)
- JWT access token с ограниченным временем жизни
- Refresh token хранится отдельно
- Возможность принудительного logout

JWT содержит:
- user_id
- global_role

JWT НЕ содержит organization_id.

—

### 2.2 Telegram Mini App

- Идентификация через Telegram ID
- Backend проверяет подпись Telegram initData
- По telegram_id определяется user
- Далее применяется стандартная RBAC логика

—

## 3. Access Scope

Backend определяет:

- Organization scope (по умолчанию)
- Global scope (для superadmin и global owner)

organization_id никогда не принимается из body запроса.

—

## 4. Разделение секретов

Используются разные ключи:

- JWT_SECRET — подпись токенов
- ENCRYPTION_KEY — шифрование чувствительных данных
- FRP_AUTH_TOKEN — токен аутентификации FRP
- TELEGRAM_BOT_TOKENS — хранятся в зашифрованном виде

Использование одного ключа для разных задач запрещено.

—

## 5. Шифрование

Шифруются:
- Telegram bot tokens
- Конфигурационные секреты (если хранятся в БД)

Пароли не шифруются, а хешируются.

—

## 6. Rate Limiting

Обязательно применяется к:

- login endpoint
- refresh endpoint
- открытию смены
- критическим операциям

Реализация:
- middleware
- Redis

Отсутствие rate limiting считается уязвимостью.

—

## 7. Проверка Telegram подписи

Mini App запросы обязаны:

1. Проверять hash initData
2. Проверять срок валидности
3. Проверять соответствие telegram_id пользователю

Запрос без проверки подписи считается небезопасным.

—

## 8. Audit Trail

Система обязана логировать:

- Изменение смен
- Расчет зарплаты
- Добавление бонусов/штрафов
- Удаление данных
- Критические изменения конфигурации

Отсутствие аудита в payroll-системе недопустимо.

—

## 9. Защита от типичных атак

Система должна предотвращать:

- IDOR
- Role escalation
- Mass assignment
- Brute force login
- Replay Telegram запросов

—

## 10. Сетевая безопасность при использовании FRP

- TLS рекомендуется завершать на VPS (Nginx/Caddy).
- FRP должен быть защищён аутентификацией (token).
- По возможности включается TLS на уровне FRP.
- Внешний доступ разрешён только к 80/443 и порту FRPS.
- PostgreSQL и Redis не публикуются наружу.
- Docker-сервисы доступны только внутри docker-сети.

—

## 11. Инварианты безопасности

- Нельзя доверять данным клиента
- Нельзя пропускать проверку роли
- Нельзя выполнять запросы без фильтрации по organization_id
- Нельзя выполнять бизнес-логику в router
- Нельзя использовать один ключ для разных механизмов

Безопасность важнее скорости разработки.
