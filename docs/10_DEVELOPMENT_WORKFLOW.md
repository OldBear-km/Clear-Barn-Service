# Development Workflow — Процесс разработки и деплоя

## 1. Общий принцип

Разработка ведётся:
- через feature-ветки
- без прямых коммитов в main
- с обязательными миграциями при изменении схемы БД
- с проверкой tenant isolation и RBAC при каждом новом endpoint

Архитектурная целостность важнее скорости.

—

## 2. Git-стратегия

Основные ветки:
- main — стабильная версия
- develop (опционально) — интеграционная ветка
- feature/* — новые функции
- fix/* — исправления
- refactor/* — изменения архитектуры

—

## 3. Работа над новой функцией

Шаги:
1. Создать ветку: feature/shifts-auto-close
2. Реализовать: модели, схемы, сервис, router, тесты
3. Добавить миграцию:
   alembic revision —autogenerate -m «...»
4. Проверить:
   - tenant isolation
   - RBAC
   - audit запись
   - логирование
5. Открыть Pull Request

—

## 4. Проверка Pull Request

Перед merge необходимо проверить:
- нет прямого обращения к БД вне сервисов
- есть фильтрация по organization_id
- проверяется роль пользователя
- нет бизнес-логики в router
- добавлены индексы при необходимости
- обновлена документация (если изменена архитектура)

—

## 5. Работа с миграциями

Любое изменение:
- таблицы
- индекса
- ограничения
- типа поля
требует миграции.

Запрещено:
- менять структуру БД вручную
- применять SQL напрямую без Alembic

—

## 6. Локальная разработка

Локально используются:
- Docker Compose
- .env файл (не коммитится)
- локальный Postgres
- локальный Redis

Перед push необходимо:
- выполнить миграции
- проверить ключевые сценарии

—

## 7. Деплой через FRP

Процесс деплоя:
1. Обновить код на локальном сервере
2. Остановить backend контейнер
3. Применить миграции: alembic upgrade head
4. Перезапустить контейнеры
5. Проверить:
   - health endpoint
   - webhook Telegram
   - FRP соединение
   - доступность домена

—

## 8. Безопасный деплой

Запрещено:
- применять миграции без бэкапа БД
- деплоить в рабочее время без необходимости
- перезапускать FRP без проверки VPS

Перед деплоем:
- создать резервную копию БД
- убедиться, что FRPS активен
- проверить сертификаты TLS

—

## 9. Тестирование

Минимально:
- unit-тесты для сервисов
- интеграционные тесты для ключевых flow:
  - открытие смены
  - автозакрытие
  - расчет зарплаты
  - добавление бонуса

—

## 10. Документация

Если меняется:
- структура БД
- роли
- логика payroll
- фоновые процессы

соответствующий .md файл обязан быть обновлен.

—

## 11. Инварианты процесса

- нет прямого push в main
- нет изменений БД без миграций
- нет endpoint без проверки ролей
- нет изменений зарплаты без audit
- нет публичного доступа к БД
