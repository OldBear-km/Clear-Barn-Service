# Audit & Logging — Аудит действий и логирование

## 1. Цель

Система обязана обеспечивать:
- прозрачность изменений данных (кто/что/когда)
- возможность разборов конфликтов по сменам и зарплате
- контроль критических процессов (автозакрытие, уведомления)
- анализ ошибок и стабильности

—

## 2. Разделение понятий

Используются два типа журналов:
1) Audit Log — история изменений бизнес-данных (не удаляется)  
2) Technical Logs — технические логи сервисов (могут ротироваться)

—

## 3. Audit Log (обязательный)

### 3.1 Где хранится
Audit лог хранится в таблице `audit_logs`.  
Удаление audit логов запрещено.

### 3.2 Когда писать audit запись
Audit должен фиксировать:
- создание/изменение/удаление смены
- автозакрытие смены
- расчет зарплаты
- ручные корректировки payroll
- добавление/изменение/удаление бонусов/штрафов
- изменения ставок и конфигурации филиала
- изменения ролей пользователей в филиале

### 3.3 Формат записи audit_logs
Поля:
- organization_id
- user_id (кто сделал действие)
- entity_type (shift, payroll_record, bonus_penalty и т.д.)
- entity_id
- action (create/update/delete/auto_close/recalc и т.п.)
- old_value (JSONB)
- new_value (JSONB)
- created_at

Важно:
- old_value/new_value хранят значимые поля
- если действие автоматическое — user_id может быть null, action помечается как system/auto

—

## 4. Правила аудита

- аудит пишется только на backend
- аудит не зависит от клиента
- аудит создается в той же транзакции, что и изменение данных
- если изменение не завершилось — audit запись не создается

—

## 5. Технические логи (application logs)

### 5.1 Формат логов
Все сервисы пишут структурированные JSON-логи.

Рекомендуемые поля:
- timestamp
- level
- module
- request_id (если есть)
- user_id (если есть)
- organization_id (если есть)
- event_type
- status
- message
- extra (json)

### 5.2 Ротация
Технические логи ротируются.  
Некритические логи удаляются через 30 дней.

—

## 6. Notification Logs

Отправка Telegram уведомлений фиксируется в `notification_logs`.

Хранится:
- кому отправлено (user_id)
- тип уведомления
- статус (sent/failed)
- текст/краткое описание
- timestamp

—

## 7. Ошибки и критические события

Критическими считаются:
- сбой автозакрытия смены
- сбой расчета payroll
- невозможность отправить обязательное уведомление
- нарушение tenant isolation (если обнаружено)

Критические ошибки:
- сохраняются отдельно
- доступны owner/superadmin
- не удаляются автоматически

—

## 8. Идемпотентность и защита от дублей

Audit и Notification не должны дублироваться при повторном запуске задач.

Для этого:
- notification_logs хранит уникальные ключи событий (например shift_id + type)
- audit_logs создается только при фактическом изменении данных

—

## 9. Инварианты

- любое изменение зарплатных данных оставляет след в audit_logs
- пользователь не может подменить автора изменений
- технические логи ротируются, audit — нет
