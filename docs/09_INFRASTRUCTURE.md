# Инфраструктура и сеть (FRP)

## 1. Цель

Система развёртывается на локальном Ubuntu Server (внутренний сервер).
Доступ из интернета обеспечивается через FRP:
- FRPS на VPS с публичным IP
- FRPC на локальном сервере

FRP используется для публикации:
- Backend API
- Web Admin
- Mini App (если требуется отдельный домен/поддомен)
- Webhook Telegram Bot (https)

—

## 2. Схема

Интернет
  ↓
DNS (A/AAAA) → VPS (FRPS)
  ↓
FRP tunnel (TCP)
  ↓
Ubuntu Server (FRPC) → Docker Compose → Nginx → сервисы

—

## 3. Компоненты

### 3.1 VPS (FRPS)
Роль: публичная точка входа.

Обязанности:
- принимать входящие соединения
- проксировать их по FRP на локальный сервер
- рекомендуется завершать TLS на VPS через Nginx/Caddy

### 3.2 Локальный сервер (FRPC)
Роль: клиент FRP, публикует локальные сервисы наружу.

Обязанности:
- поднимать docker compose
- прокидывать порты через frpc в сторону frps

—

## 4. TLS и домены

Рекомендованный вариант:
- TLS (HTTPS) завершается на VPS (Nginx/Caddy)
- далее запрос уходит по FRP до локального Nginx
- локальный Nginx раздаёт по сервисам внутри docker-сети

Это упрощает:
- выпуск сертификатов (Let’s Encrypt на VPS)
- работу Telegram webhook (нужен публичный https)

—

## 5. Список публикуемых точек

Минимально (MVP):
- api.clear-barn.online → backend
- app.clear-barn.online → frontend (web / mini app)
- (опционально) admin.clear-barn.online → web admin

Webhook бота:
- api.clear-barn.online/api/v1/telegram/webhook

—

## 6. Правила безопасности (FRP)

- FRPS слушает только необходимые порты (80/443 и порт FRP).
- FRP должен использовать аутентификацию (token).
- По возможности включается TLS на уровне FRP.
- На VPS включён firewall:
  - разрешить 80/443
  - разрешить порт FRPS только по необходимости/ограничениям
- Логи FRPS и Nginx сохраняются и ротируются.
